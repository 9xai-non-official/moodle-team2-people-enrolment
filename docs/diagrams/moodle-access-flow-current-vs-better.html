Access decision flow — Moodle today vs a cleaner design

Team 2 · people & enrolment — the answer to "can user U do action A in course C, and why?"

Today: one question, three gates, three codebases

The decision is smeared across require_login(), has_capability() in accesslib, and group checks re-implemented inside every activity. No layer explains itself.

flowchart TD
    Q["can user U do action A
in course C?"] --> ADM{"site admin?"}
    ADM -- "yes" --> BYP["full access
bypasses the whole role system"]
    ADM -- "no" --> G1["GATE 1 — require_login()
the page-level door"]
    G1 --> ENR{"active enrolment via ANY method?
(row active · instance enabled · in time window)"}
    ENR -- "yes" --> G2
    ENR -- "no" --> GST{"guest access
on this course?"}
    GST -- "no" --> X1["blocked at the door —
capability check never runs"]
    GST -- "yes" --> G2["GATE 2 — has_capability()
lib/accesslib.php"]
    G2 --> COL["collect role assignments
along the context path"]
    COL --> RES["resolve each role —
deepest override wins"]
    RES --> PRO{"any role says
PROHIBIT?"}
    PRO -- "yes" --> X2["deny — absolute veto,
crosses all roles"]
    PRO -- "no" --> ALW{"any role says
ALLOW?"}
    ALW -- "no" --> X3["deny by default"]
    ALW -- "yes" --> G3["GATE 3 — group mode,
re-checked inside each activity"]
    G3 --> GRP{"accessallgroups

    GRP -- "no" --> X4["filtered by the activity UI —
a curtain, not a wall"]

with zero explanation of why"]

Better: one engine, one verdict, one trace

Every caller hits a single policy engine. Membership is an event log projected into "effective enrolment", the enrolment requirement is declared per action, and the explanation is the response.

flowchart TD
    Q["can user U do action A
on resource R?"] --> PE["policy engine — one entry point,
every caller goes through here"]
    EV[("append-only membership events
enrolled · suspended · source-removed")] -.-> EFF
    PE --> EFF["effective-enrolment projection
one definition, computed in one place"]
    EFF --> REQ{"action declares
requires_active_enrolment?"}
    REQ -- "yes, none active" --> X1["deny + trace:
failed enrolment gate"]
    REQ -- "pass" --> PM["match policies:
principal · action · scope subtree
· group scope · conditions"]
    ADM["admins = break-glass policy,
audited — not a bypass"] -.-> PM
    PM --> DNY{"explicit DENY
matched?"}
    DNY -- "yes" --> X2["deny + trace:
policy id and scope"]
    DNY -- "no" --> ALW{"ALLOW at
deepest scope?"}
    ALW -- "no" --> X3["deny by default
+ trace"]
    ALW -- "yes" --> OK["allow + full decision trace —
the 'why' IS the API response"]
    OK --> TS["outcome events → learner transcript,
owned by the user, survives course deletion"]

What changed, and which hard case it kills

One entry pointThree scattered gates collapse into a single engine — exactly one place where a decision can be made, logged, or audited.
Declared enrolment gateEach action states requires_active_enrolment instead of depending on which PHP function happened to run first. The hidden require_login/has_capability split becomes explicit.
One DENY, two rulesPrevent/Prohibit folklore is replaced by explicit-deny-beats-allow and deeper-scope-beats-shallower. Same expressive power, published precedence, no trap.
Native group scope"Grader in course C, for group A" is one policy row. Hard case 3 stops being a fragile three-mechanism hack.
Events, not deletionsMembership is append-only and outcomes flow to a user-owned transcript. Hard case 2 becomes a replay, hard case 5 becomes possible at all.
Audited break-glassAdmin power is a policy like any other — logged and revocable — instead of a config-file bypass around the entire model.

Both charts are the simplified spine — verify every edge empirically (DB diff before/after, Check permissions as oracle) before it enters the rules catalogue. Left chart seeds how-moodle-works.md; right chart seeds my-design.md.
